Fundamental systems programming concepts demonstrated

Hash-Based Indexing

Uses hash tables for O(1) average lookup time. Handles collisions using chaining or open addressing.

JavaScript 
hash(key) => index
store[index] = {key, value, next}
üìù

Append-Only Log

Write-Ahead Log (WAL) ensures durability. All writes are appended to a log file before updating in-memory index.

Log Format 
timestamp | op | key | value
1234567890 | PUT | user:1 | Alice
‚ôªÔ∏è

Compaction

Periodically compacts log files to remove deleted/outdated entries, reclaiming disk space.

Algorithm 
1. Create new segment
2. Copy latest entries
3. Swap segments atomically
‚ö°

Memory Management

LRU cache for hot data, memory-mapped files for efficient I/O, and buffer pools for write batching.

Cache Strategy 
Cache Miss => Disk Read => Update Cache
Cache Full => Evict LRU => Insert New
üîí

Concurrency Control

Read-write locks for safe concurrent access. MVCC (Multi-Version Concurrency Control) for transactions.

Locking 
Read: Shared lock (multiple readers)
Write: Exclusive lock (one writer)
üîÑ

Crash Recovery

Replay WAL on startup to restore state. Checkpointing for faster recovery. Ensures ACID properties.

Recovery Steps 
1. Load last checkpoint
2. Replay WAL from checkpoint
3. Rebuild in-memory index

PUT Operation

JavaScript 
function put(key, value) {
  // 1. Validate input
  if (!key || value === undefined) {
    throw new Error('Invalid key/value');
  }
  
  // 2. Write to WAL
  wal.append({
    timestamp: Date.now(),
    op: 'PUT',
    key: key,
    value: value
  });
  
  // 3. Update in-memory index
  const hash = hashFunction(key);
  index[hash] = { key, value };
  
  // 4. Update cache
  cache.put(key, value);
  
  // 5. Return success
  return { success: true, latency: 1.2 };
}

GET Operation

JavaScript 
function get(key) {
  // 1. Check cache first
  if (cache.has(key)) {
    stats.cacheHits++;
    return cache.get(key);
  }
  
  // 2. Cache miss - query index
  const hash = hashFunction(key);
  const entry = index[hash];
  
  if (!entry || entry.key !== key) {
    return null; // Key not found
  }
  
  // 3. Update cache with result
  cache.put(key, entry.value);
  
  return entry.value;
}

Compaction Algorithm

JavaScript 
function compact() {
  // 1. Create new segment
  const newSegment = createSegment();
  
  // 2. Iterate through current keys
  for (const [key, value] of index) {
    if (!isDeleted(key)) {
      newSegment.append(key, value);
    }
  }
  
  // 3. Atomic swap
  atomicSwap(currentSegment, newSegment);
  
  // 4. Delete old segment
  deleteSegment(currentSegment);
  
  log('Compaction complete');
}

LRU Cache

JavaScript 
class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.cache = new Map();
  }
  
  get(key) {
    if (!this.cache.has(key)) return null;
    
    // Move to end (most recent)
    const value = this.cache.get(key);
    this.cache.delete(key);
    this.cache.set(key, value);
    return value;
  }
  
  put(key, value) {
    this.cache.delete(key);
    this.cache.set(key, value);
    
    // Evict oldest if over capacity
    if (this.cache.size > this.capacity) {
      this.cache.delete(this.cache.keys().next().value);
    }
  }
}

Optimization Techniques

  • Memory-mapped I/O for faster disk access
  • Write buffering to batch disk writes
  • Bloom filters for quick non-existence checks
  • Background compaction to avoid blocking
  • Connection pooling for concurrent clients
  • Zero-copy data transfer where possible

Trade-offs

  • Memory vs Speed: Larger cache = faster but more RAM
  • Durability vs Performance: fsync every write is slow
  • Compaction Frequency: Too often = high CPU, too rare = wasted space
  • Index Size: Full in-memory = fast but limited by RAM
  • Consistency Model: Strong consistency = lower throughput
